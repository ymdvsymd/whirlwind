# bd チケット依存関係プロトコル

チケット作成・精緻化時に適用される依存関係管理プロトコル。全 skill（moonbit-audit, log-audit, plan-to-beads, bd-detail 等）はこのプロトコルに従う。

## Step 1: 既存チケットスキャン

チケット作成・精緻化の前に、既存の active チケットを取得する:

```bash
bd list --status=open
bd list --status=in_progress
```

各既存チケットについて title と description から対象ファイル・モジュール名を抽出する。

**スキャン効率化**: 既存チケットが 20 件を超える場合は、新規チケットのキーワード（ファイル名、モジュール名、関数名）で `bd search <keyword>` を使い候補を絞る。

## Step 2: 重複検出

新規チケット候補と既存チケットを以下の基準で照合する:

| 基準 | 判定条件 |
|------|---------|
| タイトル類似 | 同一ファイル名・関数名・モジュール名を含む |
| 対象ファイル一致 | description 内の修正対象ファイルが 50% 以上重複 |
| 原因一致（bug） | 同一行・同一関数への言及 |

重複候補が見つかった場合、ユーザーに判断を求める:

```markdown
## 重複候補の確認

| 新規チケット | 既存チケット | 一致根拠 |
|------------|------------|---------|
| <new-title> | <existing-id>: <existing-title> | 対象ファイル一致: src/x.mbt |

各候補について選択してください:
- **スキップ**: 重複のため起票しない
- **依存設定**: 関連だが別問題。起票し依存関係を設定
- **独立起票**: 無関係。そのまま起票
```

重複候補がなければこのステップはスキップする。

## Step 3: 依存方向ヒューリスティクス

新規チケット間、および新規-既存チケット間の依存方向を以下の優先順位で判定する:

### 3-1. ファイル依存（最優先）

チケット A の対象ファイルがチケット B の対象ファイルから import/依存される場合、A が B をブロックする（A を先に実行）。

同一ファイルを編集する場合、より基礎的な変更（型定義、インターフェース変更）を先にする。

### 3-2. 変更種別階層

以下の階層に基づき、上位の変更が下位をブロックする:

```
インフラ/設定 > 型定義/インターフェース > 実装 > テスト > ドキュメント
```

### 3-3. 明示的順序指標

- ステップ番号（Step 1, Step 2...）→ 数字順
- "after", "requires", "前提", "〜の後" → 言及先が先行
- bug の root cause → root cause 修正が先、派生バグは後

### 3-4. 既存チケットとの方向

- 新規チケットが既存チケットと同一ファイルを対象とする場合、既存チケットが先（先着順）
- ただし新規チケットがより基礎的な変更（型定義等）の場合は新規が先

### 3-5. 判定不能

ヒューリスティクスで方向を判定できない場合、ユーザーに確認する。

## Step 4: 確認テーブルとユーザー承認

検出された依存関係を以下の形式でユーザーに提示する:

```markdown
## 依存関係の確認

| チケット | 関係 | 対象チケット | 根拠 |
|---------|------|------------|------|
| <title-A> | blocks | <title-B> | 同一ファイル: src/x.mbt |
| <title-C> | blocked-by | <existing-id> | 型定義の変更が前提 |

この依存関係を設定しますか？ (yes / 編集 / no)
```

### ユーザー応答の処理

| 応答 | アクション |
|------|----------|
| yes | `bd dep add <blocked-ticket> <blocking-ticket>` で全件設定 |
| 編集 | ユーザーの指示に従い修正後、確認テーブルを再表示 |
| no | 依存設定なし |

### 依存関係が不要な場合

全チケットが独立と判定された場合（ファイル重複なし、モジュール重複なし）、確認テーブルをスキップし、サマリーに「依存関係: なし（全チケット独立）」と記載する。

## 適用ガイドライン

- チケット 1 件のみ作成する場合でも Step 1（既存スキャン）は実施する
- `--dry-run` 時は依存分析結果をレポートに含めるが `bd dep add` は実行しない
- 過度な依存チェーンは避ける（最大深度 5 を目安とする）
- intra-batch 依存（同一セッションで作成するチケット間）と cross-existing 依存（既存チケットとの間）の両方を分析する
