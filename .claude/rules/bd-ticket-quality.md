# bd チケット品質基準

bd チケット作成・精緻化時に適用される共通品質基準。全 skill（log-audit, bd-detail, bd-runner, moonbit-audit 等）はこのルールに従う。

## 必須フィールド

- `description`: 構造化された説明（下記テンプレート準拠）
- `acceptance_criteria`: 機械検証可能な受入基準（2項目以上）
- `notes`: 補足情報（調査経緯、関連チケット、実装注意事項等）

## Description テンプレート

### bug

```markdown
## 原因
<具体的なファイル:行番号を参照して記述>

## 修正方針
1. <具体的な修正ステップ>

## 修正対象ファイル
- `<file-path>` — <変更概要>
```

### feature / task

```markdown
## 概要
<目的と背景>

## 実装方針
1. <具体的な実装ステップ>

## 対象ファイル
- `<file-path>` — <変更概要>
```

## Acceptance Criteria 要件

- 各項目は単一の、機械的に検証可能な条件
- 末尾は「こと」パターン（例: 「just test が全件パスすること」）
- 必ず「just test が全件パスすること」を含める
  - 除外: `.claude/`、`docs/`、CI 設定のみの変更
- ランタイムコード変更（`src/`, `sdk/`）の場合は「just live が全件パスすること」も含める
  - 除外: `.claude/`、`docs/`、CI 設定のみの変更
- bug: リグレッションテスト基準を含める
- feature/task: 機能の検証基準を含める

## 品質ゲート: エージェント実行可能性（P4 強制ルール）

以下の **両方** を満たさない場合、優先度を `P4`（backlog）に設定する:

| 基準 | 要件 |
|------|------|
| **how** | description に修正対象ファイル・修正方針が明確で、コード変更として表現できる |
| **AC** | acceptance_criteria が非空で、1つ以上の機械検証可能な条件を含む |

P4 降格時は notes に降格理由を記載すること。

## 依存関係の品質基準

チケット作成時は `rules/bd-dependency-protocol.md` のプロトコルに従い、依存関係を分析すること。

| 基準 | 要件 |
|------|------|
| **既存チケットスキャン** | 起票前に open/in_progress チケットとの重複・依存を確認済み |
| **intra-batch 依存** | 複数チケット同時作成時、ファイル重複チケット間に依存を設定 |
| **notes 記載** | 依存関係がある場合、notes に依存先チケット ID と理由を記載 |

## フィールド品質判定基準

| フィールド | OK | NG |
|-----------|-----|-----|
| description | 構造化セクション（`## 原因` 等）+ 具体的内容 | 空、または構造なし |
| acceptance_criteria | 2項目以上、各項目が機械検証可能 | 空、1項目未満、曖昧表現 |
| notes | 内容あり（依存関係がある場合はその記載を含む） | 空、または依存ありなのに記載なし |
| design | 内容あり（複雑なチケットで推奨） | 空（単純チケットでは不要） |
