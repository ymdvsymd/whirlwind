# bd チケット運用規約

## 計画承認後のアクション

Plan mode で計画を承認（ExitPlanMode）した直後に、次のアクションをユーザーに確認すること:

- **チケット化して実行**: `/plan-to-beads <plan-path>` でチケット起票後に実装
- **そのまま実装**: チケット化せず計画に沿って直接実装を開始

デフォルトはチケット化を提案する。ユーザーが「そのまま実装」「チケット不要」等と回答した場合はスキップする。

### チケット化フロー

1. Plan mode で計画を作成（bd チケット対応フォーマットで出力）
2. `/plan-to-beads <plan-path>` で各ステップをチケット化
3. 実装時にチケットを claim (`bd update <id> --claim`)
4. 完了したらチケットをクローズ (`bd close <id>`)

### 直接実装フロー

1. Plan mode で計画を作成
2. 計画に沿って直接実装
3. 完了後、残作業があればチケットとして起票

## /ralph-whirlwind 実行時のチケット必須化

/ralph-whirlwind で plan 実行する際は、必ず bd チケットを起票し、計画全体の内容を description に記載すること。完了時にクローズする。チケットなしでの実行は禁止。

## 結合テストでバグ発見時は bd チケットを起票

結合テスト（`just mock` / `just live`）でバグが発見された場合、各バグについて bd チケット（type=bug）を個別に起票する。

バグをその場で直すのではなく、チケットとして追跡可能にすることで、作業の可視性と優先度管理を確保する。
